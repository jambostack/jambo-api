III. Écarts détaillés — Ce qui manque à Jambo

🔴 1. Moteur d'automatisation no-code (« Flows »)

Concurrent : Directus Flows — déclencheurs (event, webhook, cron, manuel) → opérations chaînées (condition, CRUD, email, HTTP, sleep, log, notification).

Jambo : Webhooks sortants uniquement. Pas de branchement conditionnel, pas d'orchestration visuelle.

Atout Jambo : Symfony Messenger est déjà en place — la brique d'exécution async existe.

Impact : ⭐⭐⭐⭐⭐ — Fonctionnalité la plus demandée, critère de choix décisif.

🔴 2. Champs conditionnels & validation avancée

Concurrents : Directus, Payload et Strapi permettent la visibilité conditionnelle d'un champ (ex: "afficher X si champ Y = 'valeur'") et les validations custom (regex, min/max, unicité, champs calculés).

Jambo : `isRequired` + `options` par type. Pas de logique conditionnelle.

Impact : ⭐⭐⭐⭐ — Bloquant pour les schémas complexes.

🔴 3. Realtime / WebSocket / Subscriptions

Concurrent : Directus expose WebSocket + subscriptions GraphQL (push des changements).

Jambo : API REST/GraphQL en pull uniquement. Pas de canal temps réel.

Atout Jambo : Symfony a Mercure (natif) — protocole de push HTTP standardisé.

Impact : ⭐⭐⭐⭐ — Édition collaborative, dashboards live, apps mobiles réactives.

🟠 4. Live preview / Édition visuelle

Concurrents : Payload `live-preview`, Directus `visual-editing` — prévisualiser le rendu réel du front pendant l'édition.

Jambo : Pas de pont preview ↔ front.

Impact : ⭐⭐⭐ — Fort pour l'adoption marketeurs.

🟠 5. Dossiers médias + uploads résumables (TUS)

Concurrent : Directus `folders` + `tus` (reprise d'upload sur gros fichiers/coupures réseau).

Jambo : Médias à plat, pas d'arborescence ni de TUS.

Impact : ⭐⭐⭐ — Bloquant pour les grosses bibliothèques médias.

🟠 6. Dashboards / Insights (analytique intégrée)

Concurrent : Directus dashboards composés de panels (métriques, graphes, listes) sur les données.

Jambo : Aucune couche d'agrégation/visualisation.

Impact : ⭐⭐⭐ — Dashboard admin, KPIs par projet.

🟠 7. Système d'extensions / plugins

Concurrents : Les trois ont une architecture de plugins + (Directus) marketplace.

Jambo : Templates de collection/projet mais pas de mécanisme d'extension de code tiers.

Impact : ⭐⭐⭐ — Structurant pour l'écosystème à long terme.

🟡 8. Partages publics (Shares) + SAML/OIDC + SEO plugins

Écarts additionnels : liens publics temporaires, SSO fédéré, SEO/redirections, form builder, Swagger UI interactive.

IV. Forces différenciantes de Jambo

🤖

MCP natif

Aucun concurrent ne propose MCP en natif. Payload le fait en plugin, Directus via mcp-oauth. Arguments : « connectez votre CMS à Claude, Cursor, Windsurf ».

🧠

Assistant IA Studio

11 fournisseurs IA, génération de schéma, génération de contenu, agent autonome. Aucun CMS headless n'a d'équivalent intégré.

🏢

Multi-tenant natif

Ni Directus ni Strapi ne le font nativement. Payload via plugin. Arguments : agences, SaaS, isolation des projets.

🔐

2FA + Social Login

2FA admin + end-users. Social Login 4 providers (Google, Microsoft, GitHub, GitLab) aux deux niveaux. Chiffrement sodium des secrets.

📦

Export/Import projet

Export/import complet (structure, contenu, médias, end-users, settings). UUID mapping, merge conflict resolution.

⚡

Symfony 8 native

PHP performant, Doctrine ORM mature, Messenger async, Cache, Form, Security — écosystème robuste vs Node.

V. Roadmap 2026–2027

🚀 Phase 1 — Rattrapage compétitif (été 2026)
Combler les écarts les plus visibles qui bloquent l'adoption. Objectif : atteindre la parité sur les 3 fonctionnalités les plus demandées.
v1.10
Champs conditionnels & validation avancée
Effort : moyen · 2-3 jours
⭐⭐⭐⭐
🎯 ObjectifPermettre aux champs d'être visibles/masqués selon la valeur d'autres champs. Ajouter des règles de validation custom (regex, min/max, unicité, champs calculés).
📋 Spécifications
  • Visibilité conditionnelle : Field.options.conditions = [{field: "status", operator: "eq", value: "published"}]
  • Validations custom : {regex, minLength, maxLength, min, max, unique, custom}
  • Champs calculés : Field.type = "computed" avec formule
  • UI : indicateur visuel des champs conditionnels, feedback de validation en temps réel
🏗️ ImplémentationNouveau champ `Field.validationRules` JSON. Service `FieldConditionEvaluator`. Composant React `ConditionalFieldWrapper`. Validation côté serveur + client.
🎯 Concurrents visésDirectus (conditions), Payload (conditional fields), Strapi (validations)
v1.11
Dossiers médias + uploads résumables (TUS)
Effort : moyen · 3-4 jours
⭐⭐⭐⭐
🎯 ObjectifOrganiser les médias en dossiers/arborescence. Permettre les uploads résumables (TUS) pour les gros fichiers et les connexions instables.
📋 Spécifications
  • Entité `MediaFolder` (id, name, parent_id, project_id, position)
  • API CRUD dossiers : GET/POST /api/{project}/media-folders
  • Upload TUS : endpoint /api/{project}/files/tus avec protocole TUS 1.0
  • Preview/thumbnails dans l'arbre, drag & drop entre dossiers
  • Rétrocompatibilité : le dossier racine contient tous les médias existants
🏗️ ImplémentationMigration `media_folder`. Service `TusUploadHandler` (package `tus/tus-php`). Composant React `MediaBrowser` avec arbre latéral + grille.
🎯 Concurrents visésDirectus (folders + TUS)
v1.12
Realtime / WebSocket (Mercure)
Effort : moyen · 3-4 jours
⭐⭐⭐⭐
🎯 ObjectifPush temps réel des modifications de contenu vers les clients connectés. Subscriptions GraphQL + topics REST.
📋 Spécifications
  • Topics Mercure : /projects/{uuid}/content, /projects/{uuid}/media, /projects/{uuid}/comments
  • Événements émis par `MercureHubSubscriber` sur les événements Doctrine (postPersist, postUpdate, postRemove)
  • Subscriptions GraphQL via le champ subscription dans le schéma existant
  • UI : badge temps réel dans les listes (ex: "2 modifications depuis ouverture")
  • Cookie `mercureAuthorization` pour auth JWT
🏗️ ImplémentationPackage `symfony/mercure-bundle`. Service `RealtimePublisher`. EventSubscriber `MercureContentEventSubscriber`. Composant React `useRealtime` hook.
🎯 Concurrents visésDirectus WebSocket + GraphQL subscriptions
🔥 Phase 2 — Game-changer (automne 2026)
La fonctionnalité qui fera basculer les décideurs. Efforts plus élevés mais impact maximal.
v1.13
🏗️ Flows — Moteur d'automatisation no-code
Effort : élevé · 8-10 jours
⭐⭐⭐⭐⭐
🎯 ObjectifMoteur d'automatisation visuel avec déclencheurs (event, webhook, cron, manuel) enchaînés à des opérations chaînables. Le Flow Builder est le principal différenciateur manquant.
📋 Spécifications Déclencheurs :
  • Event : content.created / updated / published / deleted / comment.added
  • Webhook entrant : POST /api/{project}/flows/trigger/{flowId}
  • Cron : expression cron par flow (ex: "0 9 * * 1-5")
  • Manuel : bouton "Run" dans l'UI
Opérations :
  • Condition : if/else, switch, compare fields
  • Item CRUD : create, read, update, delete — sur les collections du projet
  • Transform : template Twig, JSONata, parse/stringify
  • Email : via SMTP du projet (déjà configuré)
  • HTTP Request : GET/POST/PUT vers URL externe
  • Sleep / Delay
  • Log : écrire dans l'audit log
  • Notification : push in-app + email
  • Webhook sortant : POST vers URL configurée
UI Flow Builder : Canvas visuel (React Flow) avec nœuds connectables, debug pas-à-pas, logs d'exécution.
🏗️ Architecture Entités : Flow, FlowStep, FlowExecution, FlowExecutionLog
Service : FlowEngine (orchestration), FlowStepRunner (par type d'opération), FlowTriggerListener
UI : assets/js/pages/Projects/Settings/Flows/ — React Flow builder
Exécution : Symfony Messenger async pour les flows event/cron, sync pour les flows manuels
Stockage : JSON structuré (étapes + connections) en base
🎯 Avantage JamboSymfony Messenger natif → exécution async robuste, retry, dead letter queue. Directus utilise un système propriétaire, Jambo peut faire mieux avec l'infrastructure existante.
🎯 Concurrents visésDirectus Flows (le seul à le faire vraiment)

  • Couverture de tests ≥ 80% (unitaires + fonctionnels)

  • Zéro bug critique ouvert depuis > 7 jours

  • Guide de migration v1.x → v2.0

  • Cache Redis systématique sur lectures publiques
  • Rate-limiting global robuste
  • Monitoring/observabilité (logs, métriques, alertes)
