Jambo est une application Symfony 8 / PHP 8.4. Les capacités ci-dessous sont déjà présentes et exploitables pour les apps et les projets.

02 Décomposition en sous-systèmes

Chaque sous-système est indépendant et suit son propre cycle spec → plan → implémentation.

2.1 · Statut : spec écrit

Jambo Sites — hébergement statique + domaine custom

  • App buildée côté navigateur (WebContainer), publiée dans var/published_sites/<uuid>/.
  • Liaison domaine ↔ projet (SiteDomain), résolution par en-tête Host (EventSubscriber), fallback SPA, anti-traversal.
  • Table de variables d'environnement par projet (WorkbenchEnvVar) injectées au build.
  • DNS + SSL gérés manuellement par le devops. Indépendant du type de serveur.
  • On garde l'export ZIP ; on supprime 1-clic (Vercel/Netlify/Railway) + Jambo Cloud Docker/Traefik.
  • Contrainte : sorties statiques uniquement (SSR hors scope → export ZIP).

Spec : docs/superpowers/specs/2026-05-29-jambo-sites-design.md

2.3 · Statut : cadré, prêt à spec

Email/SMTP par projet + Captcha anti-spam

Email — ProjectMailerSettings par projet (host/port/user/password chiffré/from), endpoint POST /api/{projectUuid}/email, service ProjectMailerService (envoi async via Messenger).

Garde-fous MVP : rate limiting (IP + projet) · mot de passe chiffré · mode (A) formulaire de contact (destinataire fixe, public protégé) — pas de relais ouvert.

Captcha stateless — gregwar est basé sur la session → problématique en cross-origin. Solution : endpoint captcha renvoyant image + token, réponse attendue en cache (usage unique), pas de session. Capacité réutilisable « à la demande » sur n'importe quel formulaire. Défense en profondeur : captcha + rate limit (+ honeypot).
2.4 · Statut : déjà disponible — rien à développer

Auth end-users avec un front statique

API REST + JWT par projet : /api/{projectId}/auth/{register,login,refresh,me,logout,forgot-password,reset-password}. Le front statique consomme l'API comme un SPA (login → JWT → Authorization: Bearer → refresh). Stateless → aucun souci de session cross-origin.

⚠️ Stockage du token : localStorage + Bearer (simple) ou cookie httpOnly Secure SameSite=None (plus sûr). MVP : Bearer + refresh court.

2.5 · Statut : analysé, à spec plus tard

Jambo Native — Twig / Turbo (SSR)

Nouveau type de template rendu côté serveur par Jambo, avec accès direct à l'EAV (zéro HTTP), SSR/SEO, sécurité, cache, i18n. La couche Host devient bi-mode (fichiers statiques ou rendu Twig).

Sécurité critique — du Twig généré par IA exécuté dans le process Jambo doit passer par le Twig Sandbox (liste blanche tags/filtres/fonctions). Mérite son propre cycle brainstorm/spec.
2.6 · Statut : nouveau, à explorer

Paiement / abonnements (gap SaaS)

Capacité serveur : intégration Stripe (ou autre), abonnements, webhooks entrants (l'infra webhooks existe déjà). Nécessaire pour un SaaS complet.

03 Peut-on développer un SaaS complet ?

Oui, en grande partie. Jambo couvre déjà le plus dur d'un backend SaaS.

Verdict : avec les capacités serveur ajoutées (email, captcha) + un module paiement/abonnements, on a un vrai SaaS. Le principal manquant est la facturation.

04 Décisions prises

  • Front statique, sans serveur Node : l'app consomme l'API Jambo. SSR hors scope (export ZIP pour l'externe).
  • Jambo sert les fichiers buildés lui-même → indépendant du type de serveur.
  • DNS & SSL hors scope code : gérés par le devops ; le domaine pointe vers le serveur Jambo.
  • On garde l'export ZIP ; on supprime 1-clic + Jambo Cloud Docker.
  • Entités : WorkbenchEnvVar, SiteDomain, WorkbenchProject.publishedAt.
  • Esthétique : structure bolt.diy + palette Jambo (émeraude sur sombre).
  • Email : mode (A) formulaire de contact en MVP.
  • Captcha : stateless (cache + token), opt-in, + rate limit.
  • Conserver l'AES-256-GCM du code Deploy pour chiffrer les creds SMTP — à confirmer.

06 Points d'attention & sécurité

  • Email — pas de relais ouvert : rate limit + (mode A) destinataire fixe.
  • Captcha cross-origin : impérativement stateless (pas de session).
  • Secrets côté front statique : toute variable injectée au build finit dans le JS livré. Les vrais secrets restent côté serveur ; isSecret ne fait que masquer l'UI.
  • JWT : choisir le stockage du token selon le besoin (XSS vs CSRF/cross-origin).
  • Twig natif : Twig Sandbox obligatoire (code généré par IA dans le process Jambo).
  • Résolution Host : chemin normalisé (realpath) confiné à var/published_sites/<uuid>/ (anti-traversal).

Hors périmètre (YAGNI)

  • SSR / runtime Node hébergé par Jambo (export ZIP pour l'externe).
  • Vérification DNS & SSL automatique (devops).
  • 1-clic Vercel/Netlify/Railway & Docker/Traefik (supprimés ; réexplorables plus tard).
  • Build côté serveur (reste dans le navigateur via WebContainer).
  • Cache/CDN, rollback de versions publiées (itération future).
